# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Датасет `S12-hw-dataset.csv`: часовой ряд `target` (4320 наблюдений), прогноз на 1 шаг вперёд.
- Разбиение по времени: 70% train, 15% validation, 15% test (индексы `[0,3024)`, `[3024,3672)`, `[3672,4320)`).
- Сравнивались B1 (`naive-last`), B2 (скользящее среднее 24 ч), B3 (`Ridge` на лагах/rolling/дне недели), R1 (`GRU` на окнах длины 48).
- Лучшая по validation MAE модель — **R1 (GRU)**; на test она оценена **один раз** после выбора.

## 2. Среда и воспроизводимость

- Python: 3.12.9
- pandas / numpy / scikit-learn: 3.0.1 / (см. окружение) / 1.8.0
- torch: 2.5.1+cu121
- Устройство (CPU/GPU): CPU (CUDA недоступна в прогоне отчёта)
- Seed: 42 (`random`, `numpy`, `torch`)

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: часовая
- Горизонт прогноза: 1 шаг
- Размер датасета: 4320
- Диапазон дат: 2025-01-01 00:00:00 — 2025-06-29 23:00:00
- Есть ли дополнительные внешние признаки: нет (только `date` и `target`)
- Комментарий (2-4 предложения): ряд демонстрирует выраженную суточную сезонность (дневные пики и ночные спады). Пропусков нет. Задача одномерного краткосрочного прогноза; наивный последний уровень даёт умеренный baseline, а нелинейная динамика выгоднее улавливается последовательной моделью.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: последовательные отрезки временной оси без перемешивания.
- Какие доли или интервалы использовались: 70% / 15% / 15% наблюдений (см. `split_summary` в `./artifacts/runs.csv`).
- Почему `random split` здесь некорректен: при случайном разбиении в обучающую выборку попадают наблюдения из «будущего» относительно части validation/test, что даёт нереалистичную оценку и допускает утечку временного контекста.

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались: `lag_1`, `lag_7`, `lag_14`
- Какие rolling-признаки использовались: `rolling_mean_7`, `rolling_std_7` (после rolling применён `shift(1)`)
- Какие календарные признаки использовались: день недели (`dow`)
- Как обрабатывались пропуски после построения лагов/rolling: строки с NaN (первые 14 шагов) исключены из обучения Ridge; для B1/B2/GRU используются исходные ряды с корректными сдвигами
- Использовалось ли масштабирование и где именно оно обучалось: `StandardScaler` на признаках Ridge — fit только на train; для GRU — `StandardScaler` на значениях `target` только на train

## 5. Модели и эксперименты (B1, B2, B3, R1)

- B1 (`naive-last`): прогноз \(\hat y_t = y_{t-1}\).
- B2 (`moving-average`): \(\hat y_t\) = среднее по 24 предыдущим наблюдениям (причинно, с `shift(1)`).
- B3 (`ridge-lag-features`): `Ridge(alpha=1)` на лагах, rolling и `dow`.
- R1 (`gru-forecast`): однослойная `GRU` (скрытое состояние 48), окно 48, выход — линейный слой к одному значению.

Дополнительно:

- Основная метрика выбора лучшей модели: **MAE на validation**
- Дополнительные метрики: RMSE, MAPE
- `window_size` для `GRU`: 48
- Batch size: 64
- Максимальное число эпох: 80
- Optimizer: Adam
- Learning rate: 0.001
- Как сохранялась лучшая модель: по минимальной val MAE в ходе эпох; сохранён `state_dict` в `./artifacts/best_gru.pt`, конфиг — `./artifacts/best_gru_config.json`

## 7. Анализ

Наивный baseline B1 использует сильную локальную автокорреляцию часового ряда и оказывается сильным простым конкурентом. Скользящее среднее B2 с окном 24 часа сглаживает суточные колебания и на validation даёт большую ошибку, так как плохо повторяет быстрые изменения уровня. Линейная модель B3 на ограниченном наборе признаков не превзошла B1: зависимость нелинейна, а часть полезного сигнала уже учтена в `lag_1`. GRU улавливает паттерны в окне из 48 часов и даёт наименьшую ошибку. Типичные утечки здесь — случайный split, «заглядывание» в будущее при rolling без сдвига, подгонка scaler на всём ряду. Выбор модели только по validation, а не по test, предотвращает оптимистичную оценку и переобучение на финальном отрезке. Ошибки на стыке режимов (например, смена уровня ряда) видны на графике факта и прогноза на test.

## 8. Итоговый вывод

Для такого часового ряда разумно начинать с наивного последнего значения и простых табличных baseline с корректными лагами, затем проверять RNN на оконном представлении. Простого Ridge на лагах достаточно только если линейная структура доминирует; здесь выиграла GRU. К нейросети имеет смысл переходить, когда baseline упирается в потолок и важны нелинейные и долгие зависимости в пределах окна. Главное по валидации временных рядов: не перемешивать время, строить признаки только из прошлого и фиксировать test для одной финальной проверки.
